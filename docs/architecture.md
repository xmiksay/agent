## Module map

| Path | Responsibility |
|---|---|

| `src/main.rs` | thin binary: builds `AppState`, runs migrations, registers routes, starts axum |
| `src/config.rs` | env ŌåÆ `Config` struct (incl. `PUBLIC_BASE_URL`, the agent's externally reachable base used to build webhook callback URLs; `NVM_DIR` and `PROJECT_DB_ADMIN_URL`/`PROJECT_DB_HOST_FOR_AGENT`, the two opt-in runtime-environment features ŌĆö see the runner section below). `log_summary` redacts the admin DSN password via `redact_db_url` |
| `src/webhook/gitlab.rs`, `src/webhook/github.rs` | per-provider HTTP handlers: HMAC verify, payload ŌåÆ `NormalizedEvent`, hand to `dispatch` |
| `src/webhook/normalized.rs` | provider-agnostic event shape (`EventKind`) |
| `src/webhook/dispatch.rs` | event ŌåÆ `TriggerReason`, dedupe, project upsert, task create. Issue events fire per the service's **trigger mode** (`assignee`\|`label`\|`both`, see below). **Issue dedup (#35):** at most one task per issue ŌĆö when a task already tracks the issue (matched by iid via `find_issue_task`), an edited/re-fired issue webhook **refreshes that task's stored title+description** (`update_issue_description`) instead of spawning a duplicate. A comment on an issue/MR with an existing agent is **delivered as a message** to that task (one agent/session per issue/MR), not a new task ŌĆö **regardless of trigger mode/label/assignee/mention**; start-gates only apply when no task exists yet. Issue/MR close stops the branch's live agent before reclaiming the worktree |
| `src/webhook/types.rs` | provider-specific payload structs |
| `src/jobs/types.rs` | `TriggerReason`, `ClaudeOutput` |
| `src/jobs/store.rs` | `TaskStore` core ŌĆö the run loop (`confirm_task` + branch-conflict guard), `update_task` (PATCH, incl. queue/priority edits), hub handle, and the **queue-admission** plumbing (`admit_lock`, `request_admit`, `run_admission_loop`). Lifecycle/CRUD/query helpers live in the sibling modules below (store.rs was split at the 400-line cap) |
| `src/jobs/lifecycle.rs` | `derive_agent_state` (read-time overlay: `is_running` ŌåÆ `running`, `is_warm` ŌåÆ `warm`, else the durable `cold\|pending\|failed`), state-name constants, the `migrate_status` backfill mapping (testable), and the durable-state transition helpers on `TaskStore` ŌĆö `recover_orphans`, `set_states`, `finish_task(agent_state, task_state, note)`, `note_task_result`, `publish_state` |
| `src/jobs/control.rs` | `TaskStore` operator controls ŌĆö `kill_task` (Pause ŌåÆ durable `cold` / `task_state` `working_on` + "paused by operator" note), `delete_task`, `continue_task` (Resume), `push_message`, `clear_pending_message` |
| `src/jobs/create.rs` | `TaskStore::create_task` (seeds `agent_state=cold` / `task_state=pending`), `update_issue_description` (rewrites an existing issue task's stored title+description in place ŌĆö the #35 dedup path), `retry_task`, issue-branch naming |
| `src/jobs/queries.rs` | `TaskStore` reads ŌĆö `list_tasks` (filters by `task_state` in SQL), `get_task`, `task_events`, `branch_diff`, `find_resumable_task_for_branch`, `find_issue_task` (existing task for an issue iid ŌĆö the #35 dedup anchor) |
| `src/jobs/queue.rs` | `TaskStore` **queue admission** (#39) ŌĆö `branch_is_live` (the shared one-agent-per-branch check, also used by `confirm_task`), `pick_next_queued` (highest-priority pending+cold+unfinished queued task, ordered `queues.priority`ŌåÆ`tasks.priority`ŌåÆage), and `try_admit_next` (under `admit_lock`, confirm queued tasks into free slots until full or dry). A distinct gate from the per-turn `Semaphore` ŌĆö see *Concurrency* below |
| `src/jobs/runner.rs` | `run_job` ŌĆö spawns the interactive stream-json session and runs the **turn loop**: per turn it takes a concurrency permit (`hub.mark_running` + `task_state ŌåÆ working_on`), forwards one operator message to the stdin writer task, waits for (and captures) that turn's `result` event, releases the permit (`hub.mark_idle` + `task_state ŌåÆ completed`, durable `cold`, no `finished_at` so the task can resume), finalizes from the captured event, then idles (warm, holding no slot) until the next message or a graceful close. Also writes the per-task `agent.env` (the agent's mutable git/gh/glab token) and spawns the agent with `BASH_ENV` pointing at it, rewriting it with a freshly-resolved token at spawn and at each turn start (#52). Also owns the dedicated stdin writer task (so control responses reach the child mid-turn) and the permission consumer that spawns `handle_permission` per request. At session end it maps the claude child exit code to **`(agent_state, task_state, note)`** ŌĆö exit 0 ŌåÆ `(cold, completed, ŌĆö)`, non-zero ŌåÆ `(failed, failed, exit detail)`, budget-kill ŌåÆ `(failed, failed, "killed: token budget")`, per-turn-timeout kill ŌåÆ `(failed, failed, "killed: per-turn timeout")` (operator Pause aborts the runner before this point; a graceful Stop makes claude exit 0 ŌåÆ completed). The per-turn watchdog (`JOB_TIMEOUT_SECS`, `0` = off) and the process-group SIGKILL + terminal-state mapping live in `src/jobs/turn_kill.rs` (`kill_process_group`, `final_disposition`); the agent is spawned as its own process-group leader so the kill reaches orphaned grandchildren (e.g. a backgrounded `cargo test`). When `NVM_DIR` is set it prepends the resolved node `bin` to the child's `PATH` (`src/jobs/nvm.rs`); when `PROJECT_DB_ADMIN_URL` is set it provisions a per-task throwaway PostgreSQL role+DB before spawn (`src/jobs/project_db.rs`), injects its DSN into the child env + initial prompt, and tears it down via a Drop guard on every exit path |
| `src/jobs/nvm.rs` | `resolve_node_bin(nvm_dir, work_dir)` ŌĆö reads the NVM filesystem directly (no shell sourcing) to resolve the node `bin` dir NVM would activate: the worktree's `.nvmrc` (trimmed) wins, else the `default` alias; a partial like `20` resolves to the highest installed `v20.*`, an uninstalled `.nvmrc` falls back to `default`. Any failure logs and yields `None` so a misconfigured NVM never blocks a run. Unit-tested against a fake NVM layout |
| `src/jobs/project_db.rs` | `ProjectDb` ŌĆö per-task throwaway PostgreSQL (#26). `provision` creates a `LOGIN`-only role + a database it owns (`agent_task_<id>`, fresh random password), `teardown` drops both idempotently (`IF EXISTS`, `WITH (FORCE)`). `ProjectDbGuard` runs teardown on Drop (spawned, since Drop can't be async); `sweep_orphans` drops every leftover `agent_task_%` at startup (the backstop for a hard SIGKILL that skipped Drop). DB-gated tests modeled on `lifecycle.rs::fresh_db` |
| `src/jobs/turn.rs` | `finalize_turn` ŌĆö per-turn bookkeeping: takes the turn's `result` event (captured off the stream), parses it, persists it, pushes commits, and posts a reply note "on demand" (only when commits landed or the turn errored) |
| `src/jobs/hub.rs` | `LiveSessions` ŌĆö per-task event hub: monotonic `seq`, `broadcast` fan-out to WS clients, and batch-persist (every 100) to the `events` table of **every** frame kind ŌĆö agent events, `auth_request`, and `status` (each consumes a seq and is persisted, not just events). Plus two back-channels to the agent ŌĆö `stdin` (operator messages, drained one per turn for pacing) and `control` (`respond_permission`, control responses written immediately). Plus per-channel in-memory liveness flags ŌĆö `is_warm` (stdin sender present) and `is_running` (an `AtomicBool` set by `mark_running`/`mark_idle` around the active turn); these two feed `derive_agent_state` (warm/running are never persisted). Plus `send_to_agent`/`stop` for routing operator messages to a live session |
| `src/jobs/prompt.rs` | `build_prompt(trigger, branch, default_branch, kind, db_note)` ŌĆö per-trigger prompt text (split out of runner). Long-lived-branch triggers (`IssueComment`/`MRComment`/`FixReview`) carry a conservative **rebase note** (`git rebase origin/<default_branch>`, abort-and-continue on conflict ŌĆö never force-resolve); the issue-opening arms (`Issue`/`IssueComment`) tell the agent to end the PR/MR description with `Closes #<iid>` so it links the change to the issue **and** auto-closes the issue on merge; comment arms tell the agent to **judge relevance** first (reply-only on small talk/questions, edit code only for real change requests). With `db_note` set, appends a paragraph telling the agent about its throwaway PostgreSQL DB (`$DATABASE_URL` + `PG*`); the DSN value itself stays in env, never the prompt text, so the password never lands in persisted `task_events` |
| `src/jobs/stream.rs` | `stream_into_entry` ŌĆö pumps a child pipe: stdout lines are published to the hub as agent events (no in-memory buffering ŌĆö `events` is the durable record), sniffing session id / output tokens via the backend, routing `can_use_tool` control requests to the permission handler instead of publishing them, and forwarding the turn's `result` event to the runner; stderr is drained to EOF and logged per-line at `debug` |

## Database

PostgreSQL via SeaORM. Migrations live in `migration/src/` and run automatically on startup (`migration::Migrator::up(&db, None)` in `main.rs`). Adding a column means a new migration file; never mutate an old one.

Tables (current set, see migration files for canonical schemas):

- `providers` ŌĆö the agent backends that can run a model. `kind` is the **system-defined** key the code maps to a CLI (`claude_code` today ŌĆö validated against `agent::backend_for`); `name` is the display label; `api_key` is **optional** and only set when the provider should run in **API mode** rather than on a subscription login (injected as `ANTHROPIC_API_KEY` at spawn, write-only across the API); `api_url` is an **optional base-URL override** for a self-hosted / Ollama-style endpoint (injected as `ANTHROPIC_BASE_URL`, not secret). The migration seeds one `claude_code` row so a fresh install can run
- `models` ŌĆö the **model catalog**: one row per runnable AI model. `provider_id` (FK ŌåÆ `providers`, `RESTRICT`) names the backend that runs it; `model_id` is the id handed to that CLI (`--model`, e.g. `claude-opus-4-8`); `alias` is the human name (`claude-mythos-5 ŌåÆ Mython`), **globally unique** (`idx_models_alias`) ŌĆö it's the operator's label, so the same `model_id` may be registered under several aliases for different settings (thinking on/off, unbound, distinct pricing). A price table (USD per **1M** tokens: `input_price`, `output_price`, `cache_write_price`, `cache_read_price`) and optional run settings (`thinking` ŌĆö a **nullable** bool: `null` = the model's own default, else force on/off; `effort` text) ride along. At most one row is the global default (`is_default`, enforced by a partial unique index). **`unbound` (dangerous, default false):** when set, a task on this model runs with `--dangerously-skip-permissions` ŌĆö every tool call (incl. arbitrary Bash) runs with no allowlist check and no operator approval; the SPA flags it in red wherever the model appears. **Model resolution** at run time is `tasks.model_id ?? global default ?? the CLI's own default` (the service per-type mapping is resolved earlier, at task creation) ŌĆö a deleted model row never blocks a run, it just falls through
- `service_models` ŌĆö per-service, per-**trigger type** model selection (`service_id`, `trigger_type`, `model_id`), unique on `(service_id, trigger_type)`, both FKs `CASCADE`. A service can run `review_mr` on one model and `issue_comment` on another. `tasks.model_id` (nullable FK ŌåÆ `models`, `SET NULL`) is the model that task runs ŌĆö seeded at creation from the owning service's mapping for the task's trigger type (else the global default), operator-overridable via PATCH while pending
- `queues` ŌĆö the **task queue** catalog (#39): `id`, `name`, `priority` (smallint, queue-vs-queue ordering, higher = sooner), `created_at`. An operator-orderable backlog bucket; the migration seeds one `Default` queue (stable UUID) so a fresh install can enqueue with no setup. The scheduling logic lives in `jobs/queue.rs`
- `tasks` ŌĆö one row per work item, with **two orthogonal state axes**. `task_state` (persisted, operator-owned lifecycle: `pending|working_on|completed|failed`) is auto-advanced by the runner but freely PATCH-overridable. `agent_state` is **derived at read time** by `derive_agent_state` (`cold|warm|pending|running|failed`) ŌĆö its persisted backing column narrows to `cold|pending|failed`, and `warm`/`running` are overlaid from the hub's in-memory liveness flags, never stored. There is no `killed` state ŌĆö the disposition reason (paused / budget / orphan) is recorded as `task_sessions` text instead. `pending_message` carries a queued follow-up for the resume path. **Normalized:** a task carries only `project_id` (NOT NULL, `ON DELETE CASCADE`) plus `branch` and the working `session_id`; remote URL, default branch, provider and owning service are resolved at run time through `project_id ŌåÆ projects ŌåÆ service` (not duplicated on the row), and the API re-derives them onto `TaskView` so the SPA still sees them. `model_id` (nullable FK ŌåÆ `models`) is the task's selected model (see `service_models` above). **Queue link (#39):** `queue_id` (nullable FK ŌåÆ `queues`, `ON DELETE SET NULL` ŌĆö deleting a queue un-queues its tasks) + `priority` (smallint, default 0, in-queue sort key); indexed `(queue_id, priority)`. The scheduler only pulls tasks with `queue_id` set; both are operator-editable via PATCH **while pending**. "Filter out completed" is the `task_state = pending` predicate the picker uses
- `events` ŌĆö durable hub-frame stream; one row per frame of any kind (agent event, `auth_request`, `status`) distinguished by the `kind` column, PK `(task_id, seq)`, each frame consuming a unique seq, append-only, batch-inserted (100 at a time) by the live hub, cascade-deleted with the task. **Turns are derived, not stored:** the SPA folds this flat stream into per-turn input/output (`useTurns.ts`) from the `type:"user"` (input, via `--replay-user-messages`) and `type:"result"` (output) markers already present ŌĆö no `turn_index` column, no turns table, no second writer in the hot path
- `task_sessions` ŌĆö per-run cost / turns / tokens / result text + the run's claude `session_id` and `created_at`. **1:N with tasks**: turns within one agent run (same `session_id`) accumulate into a single row; a new run (fresh `session_id` ŌĆö a resume after kill, or a retry) starts a new row, so run history is preserved. `get_task` / the detail view surface the most recent row
- `projects` ŌĆö discovered repos, per-project `allowed_operations` glob list and `env_file` (a `.env`-style minijinja template injected as env vars at agent spawn)
- `project_branches` ŌĆö branches the agent has touched, with `issue_iid` / `pr_iid` linkage and status
- `auth_requests` ŌĆö operator-approval items raised by the in-process permission handler
- `service` ŌĆö provider config: kind, base URL, bot username, token, webhook secret, `autofire` (when true, a newly-created task from this service's webhook is auto-confirmed ŌĆö started running immediately instead of left pending for a manual confirm). **Trigger mode:** `trigger_mode` (`assignee`\|`label`\|`both`, default `assignee`) + `trigger_label` (default `''`) decide which issue events fire the agent ŌĆö `assignee` matches when the bot is an assignee, `label` matches when the issue carries `trigger_label`, `both` fires on either. Label mode is how a GitHub App (which can't be an issue assignee) gets triggered. Both are read-only on `ServiceView` and writable via `NewService`/`UpdateService`. **Per-task-type models:** the service's `service_models` mapping (`trigger_type ŌåÆ model_id`, surfaced as `ServiceView.models` and set via the `models` field on create/update) chooses which catalog model a new task inherits at creation, by trigger type; an unmapped type falls back to the global default. the credential is a **type + value** pair: `auth_kind` (`pat`\|`app`, default `pat`) is the type, and `app_credentials` (JSONB, null for `pat`) is the value ŌĆö the provider-specific secret bundle (GitHub App: `{app_id, private_key, installation_id}`). **GitHub App (#9) is wired**: for `auth_kind = 'app'` GitHub services, `resolve_token` mints installation tokens from the bundle (see *GitHub App auth* below); `installation_id` is optional at create time ŌĆö the install callback fills it in (see [`docs/application-integration.md`](application-integration.md)). GitLab has no `app` flow. **GitLab uses `pat` with its own bot identity (#10)** ŌĆö a **Group/Project Access Token** mints a `group_NNN_bot_*` service account, so the agent acts as itself, never the operator. **Required token scopes** (`auth_kind = 'pat'`, the token does git transport, note posting, and webhook registration): GitHub needs `repo` + `admin:repo_hook`; GitLab needs `api` + `write_repository` and Maintainer/Owner on the group/project. A GitLab Group/Project Access Token expires Ōēż365 days ŌĆö rotate it (manually, or via the access-tokens API) before expiry. GitHub App services need no PAT and no `admin:repo_hook` ŌĆö permissions come from the App install and the webhook is app-level.

## Workspace layout on disk

```
$REPO_BASE_PATH/
ŌööŌöĆŌöĆ <service_slug>/
    ŌööŌöĆŌöĆ <project_slug>/
        Ōö£ŌöĆŌöĆ <branch_slug>.lock           # advisory file lock per branch
        Ōö£ŌöĆŌöĆ <branch_slug>/               # one git worktree per branch
        Ōöé   ŌööŌöĆŌöĆ .git/
        ŌööŌöĆŌöĆ ŌĆ”
```

`slugify` lower-cases and replaces non-alphanumerics with `__`. Each task confirms the worktree exists (clone/fetch, preserving an already-checked-out branch), then runs `claude --print [--resume <sid>] [--model <id>] --input-format stream-json --output-format stream-json --verbose --replay-user-messages --permission-mode default --permission-prompt-tool stdio` (`--model` present only when a catalog model is resolved for the task; if that model's provider carries an `api_key`/`api_url`, `ANTHROPIC_API_KEY`/`ANTHROPIC_BASE_URL` are set in the child env for API-mode billing against the chosen endpoint). **If the resolved model is `unbound`, those two permission flags are replaced by `--dangerously-skip-permissions`** ŌĆö no control-protocol prompts are emitted and every tool runs unchecked. The agent's own config/hook files are **not** written into the worktree ŌĆö permission behavior is a CLI flag, and tool gating is the in-process control-protocol handler.

The process is **long-lived and turn-based**. The runner's turn loop, per turn: takes a `MAX_CONCURRENT_JOBS` permit (`hub.mark_running` ŌåÆ derived `agent_state` `running`, `task_state ŌåÆ working_on`), writes one operator message to stdin (the first turn uses the initial prompt), waits for that turn's `result`, **releases the permit** (`hub.mark_idle` ŌåÆ derived `agent_state` `warm`, `task_state ŌåÆ completed`), finalizes (push commits + reply-on-demand), then **idles warm** ŌĆö the agent process stays alive but holds **no concurrency slot** (an idle agent is not a running agent). A new message (live via `hub.send_to_agent`, or a resume) wakes it into the next turn. The session ends on **stop** (hub drops the stdin sender ŌåÆ EOF ŌåÆ graceful exit), **pause** (SIGKILL), token-budget kill, or issue/MR close. `push_message` delivers to a warm agent first (`hub.is_warm`) and only resumes the session when there's no live agent ŌĆö so a follow-up never spawns a second agent on the same branch.

**Two distinct concurrency gates (#39).** `MAX_CONCURRENT_JOBS` is used by *two* independent mechanisms that must not be conflated:

1. **Per-turn `Semaphore`** (`store.rs`) ŌĆö a permit is taken/released **inside each turn** (`mark_running`/`mark_idle`). It throttles *actively-processing turns*; a warm idle agent holds **no** permit. This is back-pressure on compute, not on task lifetimes.
2. **Task admission** (`jobs/queue.rs`) ŌĆö a task is *admitted* while its `run_job` future is alive (warm-idle, running, or spawn-in-flight), counted by `RunningTasks::len()`. Free slots = `MAX_CONCURRENT_JOBS ŌłÆ len()`. Here a warm idle agent **does** count, because its branch is reserved.

Confirmed work is **not** spawned eagerly when a queue is involved: a task is enqueued (`tasks.queue_id` set, `task_state = pending`), and `try_admit_next` pulls the highest-priority pending queued task into a free admission slot when one opens. A slot opens when a run future ends (the spawned closure calls `request_admit`, waking `run_admission_loop` ŌĆö it can't call `try_admit_next` directly, as that would close a `confirm_task ŌåÆ try_admit_next ŌåÆ confirm_task` async cycle the `tokio::spawn` `Send` bound rejects), on operator **pause**/**delete** (which abort the future, so they signal admission themselves), on an **enqueue/reprioritize** PATCH, and at **boot** (a drain after `recover_orphans`). The `admit_lock` is held across the slot-count read and `confirm_task` so two concurrent completions can't both claim the last slot. **Manual Run (`/confirm`) and autofire bypass queueing** ŌĆö they confirm immediately regardless of queue order, as an operator override.

**Branch selection (a task never runs on the default branch).** `TaskStore::create_task` derives and persists `tasks.branch`: MR triggers reuse the MR's `source_branch`; an `Issue` trigger derives `<iid>-<slug(title)>` (e.g. `42-fix-login-button`); an `IssueComment` reuses the branch the original issue task recorded (`find_branch_for_issue`), falling back to bare `<iid>`. **Comments delegate:** for a `NoteAdded` event the dispatcher first locates a resumable task straight from the note's target (`resumable_task_for_note` ŌåÆ MR `source_branch` / issue iid ŌåÆ branch ŌåÆ `find_resumable_task_for_branch`) **before any trigger gate**. If one exists it delivers the comment via `push_message` to that one task ŌĆö continuing the same agent/session ŌĆö **regardless of trigger mode/label/assignee/mention**. Only when no task exists yet do the NoteAdded start-gates in `build_trigger` (mention / assignee) decide whether a bare comment may spin up a fresh task. Dedupe (`note_event_id` ŌåÆ same key a comment `TriggerReason` would produce) is preserved on both the reattach and the fresh-start path. `workspace::git::clone_or_fetch(path, auth, branch, default_branch)` clones on first use, fetches, then: if the worktree is **already on `branch`** it is left untouched (local commits + uncommitted work are preserved across runs ŌĆö what makes resume-on-message safe); otherwise it checks out `origin/<branch>` if it exists remotely, else creates the branch from `origin/<default_branch>` (`git checkout -B`, no force-reset). `push_changes` uses `git push -u origin HEAD` so a fresh branch gets its upstream. `run_job` hard-`bail!`s if the resolved branch equals the default branch.

**Git transport is token-HTTPS, not host SSH.** `HttpsAuth::from_remote_url` turns the project's `remote_url` ŌĆö accepting either an SSH form (`git@host:path.git`, `ssh://ŌĆ”`) or an HTTPS form (`https://host/path.git`, any embedded userinfo stripped), detected from the URL ŌĆö into a secret-free HTTPS remote (`https://host/path.git`) plus a git `credential.helper` that reads the token from an env var at call time. `clone_or_fetch` clones with that helper, then persists the clean remote + helper into the worktree's `.git/config` (also migrating any pre-existing SSH worktree). The token is never written into `.git/config` ŌĆö it lives only in the process environment of each git child (`GH_TOKEN`/`GITLAB_TOKEN`), so the runner's `push_changes`, **and the agent's own `git push -u origin HEAD`**, both authenticate transparently without a host SSH key.

The spawned `claude` inherits a provider-scoped token env var so `gh`/`glab` and `git` inside the worktree authenticate against the same token used for clone + note posting: `GH_TOKEN` for GitHub services, `GITLAB_TOKEN` for GitLab services. The value comes from `provider::credentials::resolve_token(service.credentials()?)` ŌĆö the stored PAT for `pat` services, or a freshly-minted GitHub App installation token for `app` services (GitHub installation tokens authenticate git-over-HTTPS as `x-access-token:<token>`, exactly the username `HttpsAuth` already uses). The project's `env_file` is rendered (minijinja, against runtime vars `branch`/`default_branch`/`url`/`project`/`service`/`task_id` ŌĆö see `src/project/env.rs`) and applied **before** this reserved var, so a project env can never clobber the PAT. The prompt (`src/jobs/prompt.rs`) also tells the agent the CLI is pre-authenticated and the existing `origin` pushes over HTTPS, so it must not add SSH keys, set tokens, or rewrite the remote.

**Opt-in runtime-environment features (#26).** Two independent, separately-toggleable features change *what environment* the spawned `claude` runs with. Both are disabled by default ŌĆö with neither var set the spawn `Command` is byte-for-byte the prior behavior.

- **NVM (`NVM_DIR`).** When set, the runner resolves the node toolchain NVM would activate by reading the NVM filesystem directly (`src/jobs/nvm.rs::resolve_node_bin`, no shell sourcing) ŌĆö the worktree's `.nvmrc` wins, else the `default` alias ŌĆö and prepends `<NVM_DIR>/versions/node/<version>/bin` to the child's `PATH`. The child inherits the process env, so only `PATH` is rewritten. Any resolution failure logs a `warn!` and leaves `PATH` unchanged (a misconfigured NVM never blocks a run).
- **Per-task PostgreSQL (`PROJECT_DB_ADMIN_URL`, optional `PROJECT_DB_HOST_FOR_AGENT`).** When set, the runner provisions a **throwaway `LOGIN`-only role + a database it owns** (`agent_task_<task_id>`, fresh random password) at session start ŌĆö once per task, not per turn, so data survives across turns ŌĆö via the admin DSN's `CREATE ROLE`/`CREATE DATABASE` privilege. It injects the connection into the child env **after** the project's rendered `.env` so a project can't clobber it: `DATABASE_URL` (the full DSN) plus `PGHOST`/`PGPORT`/`PGDATABASE`/`PGUSER`/`PGPASSWORD` so bare `psql` connects. The agent-facing host defaults to the admin URL's host, overridable via `PROJECT_DB_HOST_FOR_AGENT` for containerized setups. `build_prompt` is told to append a paragraph pointing the agent at `$DATABASE_URL` ŌĆö **the DSN value stays in env, never the prompt text**, so the password never lands in persisted `task_events`. Teardown (`DROP DATABASE ŌĆ” WITH (FORCE)` + `DROP ROLE`, both `IF EXISTS`, idempotent) is **guaranteed on every exit path** by a `ProjectDbGuard` whose `Drop` spawns it (covers `?`, graceful end, Pause-abort); a hard SIGKILL that skips Drop is mopped up by `ProjectDb::sweep_orphans` at startup (`main.rs`, after `recover_orphans`), which drops every leftover `agent_task_%` object ŌĆö trivially safe because no in-flight task survives a restart. The DB is least-privilege (the role owns only its own DB; no `CREATEDB`/`CREATEROLE`/`SUPERUSER`) and unrelated to the agent's own control-plane `DATABASE_URL`.

**The agent's own token is mutable, via `agent.env` + `BASH_ENV` (#52).** The frozen spawn-time env var alone can't be rotated, and an App token lives only ~1h ŌĆö so a warm-idle session or a single >1h turn would leave the agent's *own* `git`/`gh`/`glab` calls using a dead token. The runner therefore also writes `<work_dir>/.git/agent.env` (`export GH_TOKEN='ŌĆ”'`, `0600`, atomic temp+rename ŌĆö `src/workspace/token_env.rs`) and spawns the agent with `BASH_ENV` pointing at it. Bash sources `$BASH_ENV` at the start of every non-interactive shell (how the CLI's Bash tool runs commands), so each invocation re-reads the **current** token ŌĆö the credential helper still reads `$GH_TOKEN`, and `gh`/`glab` read it too, so one mechanism covers `git + gh/glab`. The file lives under `.git/` so it never enters the working tree, the diff, or a commit. The path is fixed for the process; only the contents change ŌĆö the runner rewrites them with a freshly-resolved token at **spawn** (including stop/pauseŌåÆresume, which re-spawns) and at the **start of every turn**, so a warm-idle wake-up is fresh with no operator action. The direct `.env(provider_token_var, ŌĆ”)` stays as the initial value; the sourced file wins inside each subshell. For a single turn that outlives the TTL mid-flight (env can't auto-refresh mid-turn), `POST /api/tasks/{id}/token` rewrites the file on demand (optionally with an operator-supplied token) ŌĆö the SPA's "Refresh token" button.

**Mid-session token refresh (#44).** A GitHub App installation token lives ~1h, but a turn-based session can outlive that. The runner therefore keeps the credential *source* (`service.credentials()`), not just the start-of-session string, and `finalize_turn` ŌåÆ `push_changes` calls `resolve_token` **just-in-time before every push** ŌĆö cheap for a PAT (returns the stored token), and for an App it transparently re-mints once the cached token enters its 5-min refresh margin. So the runner's per-turn push always uses a live token. The `claude` child's *process* env (`GH_TOKEN`/`GITLAB_TOKEN`) is still captured once at spawn and cannot be updated in place ŌĆö but the agent's own commands no longer rely on it: they read the token from the mutable `agent.env` the runner rewrites per turn (#52, above), so even a `git push`/`gh` late in a long session uses a live token. And the runner still re-pushes the branch (`git push -u origin HEAD` over `@{u}..HEAD`) at the end of every turn with a freshly-resolved token as a backstop, so the agent's local commits land regardless. The session-start `clone_or_fetch` is unchanged ŌĆö it already runs with a fresh token.

## Webhook auto-registration

When `dispatch` upserts a project for the **first time**, it spawns a best-effort, idempotent webhook registration (`GitProvider::ensure_webhook`) so operators no longer wire hooks by hand. The callback URL is `{PUBLIC_BASE_URL}/webhook/{kind}/{slug}` and the body sets the service's `webhook_secret` plus the issue/MR/comment events the dispatcher acts on; SSL verification is on. Registration is idempotent ŌĆö it lists existing hooks and updates the one already pointing at our `/webhook/ŌĆ”` path (so a changed `PUBLIC_BASE_URL` migrates in place) rather than duplicating. It runs in the background so it never delays or fails the inbound delivery, and is a logged no-op when `PUBLIC_BASE_URL` is unset. **App-auth services skip per-repo registration entirely** ŌĆö a GitHub App has a single app-level webhook (configured on the App, delivering every installed repo's events to one URL ŌåÆ `ŌĆ”/webhook/github/{slug}`), so there is no per-repo hook to create and the App token isn't scoped to do so. The inbound HMAC check (`verify_signature`) is unchanged; the secret is just set at the app level.

## GitHub App auth

For `auth_kind = 'app'` GitHub services, `resolve_token` mints an **installation access token** instead of returning a PAT (`src/provider/github/app.rs`):

1. **JWT** ŌĆö sign a short-lived RS256 JWT with the stored `private_key`: `iss = app_id`, `iat = now-60s` (skew), `exp = now+9min` (under GitHub's 10-min cap).
2. **Exchange** ŌĆö `POST {api_base}/app/installations/{installation_id}/access_tokens` with the JWT ŌåÆ `{ token, expires_at }`, a ~1h installation-scoped token. `api_base` is threaded in from the service's `base_url`, so GHES works unchanged.
3. **Cache** ŌĆö the token is held in-process (keyed by `{api_base}#{installation_id}`) and reused until ~5 min before `expires_at`, so the per-request `resolve_token` callers don't mint on every call.

`installation_id` is blank until the App is installed; while blank, `resolve_token` fails with a clear "not installed yet" message rather than minting.

**Install flow.** The operator creates the service with `app_id` + `private_key` (no `installation_id` yet), then clicks **Install GitHub App** in the SPA. That calls `GET /api/services/{id}/github_app/install`, which looks up the App's public URL (`GET /app`, JWT-authed) and returns `{ install_url }` = `{html_url}/installations/new?state={service_id}`. The SPA navigates there; after the operator grants repos, GitHub redirects the browser to the App's configured setup URL ŌåÆ `{PUBLIC_BASE_URL}/github_app/callback?installation_id=ŌĆ”&state={service_id}`. That public handler writes `installation_id` into the service's `app_credentials` and bounces back to the SPA. The `ServiceView.app_installed` flag reflects whether an `installation_id` is on record.

## How an operator approval works

Tool gating runs entirely over the stream-json **control protocol** ŌĆö the same stdin/stdout the runner already owns ŌĆö so there is no hook script, no `/internal/authcheck` loopback, and no `CLAUDE_TASK_ID`/`AGENT_PORT`. `--permission-mode default --permission-prompt-tool stdio` makes the CLI emit a `can_use_tool` `control_request` on stdout for every non-trivially-safe tool and wait for a `control_response` on stdin.

1. The stdout pump (`src/jobs/stream.rs`) detects a `can_use_tool` line via `backend.parse_permission_request` and forwards it to the permission consumer instead of publishing it as a timeline event.
2. `handle_permission` (`src/jobs/permission.rs`) applies the policy: any tool other than `Bash`/`AskUserQuestion` (edits, reads) is **allowed immediately**, echoing the input back as `updatedInput`.
3. For `Bash`, the command is matched against the project's `allowed_operations` glob list (`auth/operations.rs`). On hit it allows immediately.
4. On miss (or for `AskUserQuestion`), it creates an `auth_requests` row, **publishes it to the task's live hub** (so the task page shows the pending approval instantly over the WS), and parks on `AuthWaiter.register(id).notified()`. The wait honors `OPERATOR_APPROVAL_TIMEOUT_SECS`: `0` (default) blocks **indefinitely** (never auto-deny); `>0` wraps the wait in a timeout that, on expiry, auto-denies via `resolve_and_publish` (statusŌåÆdenied + resolution frame) so the row leaves `pending` instead of being re-surfaced forever (#45). The notified future is `enable()`d before the await to close the registerŌåÆawait race a never-timing-out wait would otherwise expose.
   **Concurrency note:** the per-turn `MAX_CONCURRENT_JOBS` permit is held by the runner's turn loop while the agent is blocked on a `can_use_tool` response (no `result` event arrives until the tool resolves). So an indefinite approval wait keeps its permit ŌĆö enough simultaneous parked approvals can starve other tasks' turns. The permission handler itself runs off the turn loop (spawned per request), but it can't release the turn's permit.
   The SPA also coalesces all pending asks into a single browser notification (`frontend/src/notifications.ts`) that links to the auth queue, so a backgrounded operator is alerted without per-ask spam.
5. The operator resolves via `POST /api/auth_requests/{id}/resolve` (or `bulk_resolve` for many at once ŌĆö the queue's "Deny all"). Both go through `resolve_and_publish`: the store wakes the waiter and publishes the resolution to the hub; the single-resolve handler then encodes the decision and `hub.respond_permission` writes the `control_response` straight to the agent's stdin, bypassing the per-turn pacing so a mid-turn prompt is answered without waiting for the turn to end.
   - **Bash** approvals encode `allow` (echoing the input back as `updatedInput`) or `deny` with the optional reply as the message.
   - **AskUserQuestion** approvals take the headless **allow + answers** path (#48): the resolve body carries a structured `answers` object (`{ "<question text>": "<chosen label | custom string>" | ["<label>", ŌĆ”] }`) which `resolve_auth_request` stringifies into `operator_reply` ŌĆö that column **doubles as the answers carrier** for question approvals. `map_decision` parses it back and emits `allow` with `updatedInput = { questions: <original questions>, answers: <answers> }`, so the CLI records a normal (non-error) tool result and an arbitrary custom string is honored (the built-in "Other" path). A decline, or an approval whose carrier isn't a parseable answers object, falls back to `deny` with the reply as the message. The SPA's `AuthApprovalForm.vue` renders the structured per-question form (single/multi choice + a per-question "Other (custom answer)" text input) and requires every question answered before enabling **Approve**.
